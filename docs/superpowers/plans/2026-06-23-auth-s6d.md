# Auth S6-D — CLI device approval — implementation plan

Design: `docs/superpowers/specs/2026-06-23-auth-s6d-device-approval-design.md`.
Sequenced so the **login loop is closeable + testable on desktop early** (Phase
1), then the camera/deep-link platform work layers on (Phase 2). geo/ASN +
keychain + E2E are out (deferred).

## Phase 1 — close the loop (platform-agnostic; enter-code path)

1. **API `GET /device/pending`** — `apps/api/src/auth/device.ts` (lookup helper)
   + route in `apps/api/src/app.ts`. Session-auth, shared `account:approve:<sub>`
   rate-limit + lockout, uniform 404, audit `device.lookup`. Regenerate the
   Vercel bundle (`apps/api/api/index.js`). **Tests:** `apps/api/test/
   device-pending.test.ts` — hit/miss/expired/lockout/401/audit.
2. **AuthClient** — `devicePending` / `deviceApprove` / `deviceDeny` + `PendingDevice`
   in `AuthClient.kt`; MockEngine tests (`AuthClientTest.kt`).
3. **State/actions/reducer** — `Route.EnterCode|AuthorizeDevice`, `pendingDevice`/
   `deviceBusy`/`deviceError`, the device actions, reducer arms. `ReducerTest`
   cases incl. expired/denied/failed.
4. **AuthEngine** — `lookupDevice`/`approveDevice`/`denyDevice` (mutex +
   `callWithRefresh`); `AuthEngineTest`.
5. **Screens** — `DeviceApprovalScreens.kt`: `EnterCodeScreen`,
   `AuthorizeDeviceScreen` (owner-family selector via `whoami.families`
   role=owner; static row when 1), `DeviceDeniedScreen`, `DeviceExpiredScreen`.
   Snapshot tests vs the A8b mockup PNGs (light+dark).
6. **FeedApp + entry** — two `when(route)` arms; CTA wired from `FamilyNullState`
   + a "Connect a device" affordance in `DevicesScreen`.
7. **CLI QR** — `zxing-core` in `apps/cli/build.gradle.kts`; render
   `verification_uri_complete` in `deviceLogin` above the text code; keep text.
   `./gradlew build`.
8. **E2E round-trip (desktop)** — dev-auth API: `dayfold login` → enter code →
   approve → "logged in" → `dayfold push` 200. Verify via `[redux]` log.

**Phase-1 gate:** loop closes via typed code on desktop + Android; suites green.

## Phase 2 — scan + deep-link (platform-specific; needs the viewfinder mockup)

9. **Scan/deep-link mockups** (ADR 0008 design-first) → operator sign-off — blocks
   10. Brief: `designs/DESIGN-BRIEF-device-scan.md`. Authored with the
   `frontend-design` skill (or the `claude_design` MCP connector against the shared
   project), extending `designs/.../Auth-Phone.dc.html`. Views: `scandevice`
   (camera viewfinder), `scanprimer` (permission rationale), `scandenied`
   (permission denied → fallback), `deviceresume` (deep-link cold-start handoff),
   and an updated `entercode` with a "Scan QR code" affordance — each light + dark,
   mounted in `Auth.dc.html`.
10. **QR scanner `expect/actual`** — `QrScanner` + `qrScanSupported`. Android
    (CameraX + ML Kit `barcode-scanning`, CAMERA permission); iOS (AVFoundation
    via `UIKitView`); desktop unsupported. Parse URL→`user_code`. Scan button on
    `EnterCodeScreen` shown only when `qrScanSupported`.
11. **Deep-link** — Android App Links (`autoVerify` intent-filter, API-origin
    host, `/device`) + `/.well-known/assetlinks.json` on Vercel; iOS Universal
    Links (associated-domains entitlement) + `apple-app-site-association` on
    Vercel. Open handler: signed-in → lookup→AuthorizeDevice; else stash code →
    onboarding → **resume**. Update the static `/device` page copy.
12. **E2E (Android)** — scan the terminal QR → AuthorizeDevice → approve →
    `dayfold push` 200; deep-link cold-start resume verified.

## Escalations / dependencies
- **geo/ASN datacenter warning** — deferred (vendor + cost); slice 2 of *this*
  feature family if/when approved.
- **Scan/deep-link mockups + sign-off** (`designs/DESIGN-BRIEF-device-scan.md`) —
  gates step 10. Authoring via the `claude_design` MCP connector requires the
  connector be authorized (`/design-login` → `user:design:read/write`).
- **`zxing-core`** (CLI) + **CameraX/ML Kit** (Android) + **AVFoundation** (iOS)
  dependencies — minor maintenance-burden adds; ML Kit barcode is on-device (no
  network, no cost).

## Risks
- Deep-link verification (`assetlinks.json`/AASA) is fiddly to verify in CI —
  manual device check + keep enter-code as the guaranteed fallback.
- iOS camera via `UIKitView` interop is the highest-effort actual — Android +
  enter-code can ship first if iOS lags (desktop already degrades to enter-code).
- Cold-install resume (deep-link before sign-in) is the subtle path — explicit
  reducer test for stash→onboard→resume.
